# 功能评估框架

| 文档状态 | Phase 2 Bible draft |
| --- | --- |
| 版本 | 2.0.0-draft |
| 负责人 | FutureOS product, AI, design, banking, and engineering teams |
| 最后更新 | 2026-07-13 |

## 目的
提供用于接受、修订或拒绝 FutureOS 功能的评分框架。

## 范围
适用于新想法、Phase 2 工作流、demo 功能、AI 行为、产品流程和银行执行能力。

## 产品信念
功能质量应由结果价值、信任、解释性、OCBC 优势和长期有用性判断，而不是只看新奇。

## 必须出现的行为
- 每个拟议功能必须按所有标准从 0 到 5 评分。
- 记录假设、缺失证据和所需决策记录。
- 即使视觉上很惊艳，只要削弱同意、解释性或架构，也必须拒绝。

## 禁止出现的行为
- 功能不能只因视觉好看就通过。
- AI 功能不能解释自己就不能通过。
- 银行功能没有目标连接就不能通过。
- 自主功能没有 guardrail 就不能通过。

## 设计影响
- 高分功能应在 UI 中容易理解。
- 低解释性需要先重新设计。

## AI 影响
- AI 原生价值衡量 AI 推理是否必要。
- 数据可用性和信心必须明确。

## 银行业务影响
- OCBC 特定价值和监管可行性是不同标准。
- 功能可能战略上有价值，但当前不可行。

## 工程影响
- 技术可行性和可扩展性包括可维护性和审计能力。
- 不能假设数据可用。

## 示例
- Guardian Memory 在关系、长期有用性和差异化上得分高。
- 通用 AI chat 得分低，因为它重复聊天机器人行为并削弱架构。

## 反模式
- 只按 demo 戏剧性选择功能。
- 没有适配理由就接受产品推荐。
- 把需要研究的功能说成已存在。

## 决策检查清单
- [ ] 评分是否记录？
- [ ] 低分项是否处理？
- [ ] 是否需要决策记录？
- [ ] 同意是否清楚？
- [ ] 是否由一个核心系统拥有？

## Phase 2 影响
- 用此框架在更深模拟、Guardian Memory、产品编排和自主功能之间选择。

## 未来演进
- 评分标准可以演进，但用户控制、信任和人生结果影响应保持核心。

## 修订说明
- 评分是决策辅助，不替代判断。


## 评分标准
| 标准 | 0 分含义 | 5 分含义 |
| --- |--- |--- |
| 用户问题严重度 | 没有清楚问题。 | 痛苦、频繁或高风险问题。 |
| AI 原生价值 | AI 是装饰。 | AI 推理是必要的。 |
| OCBC 特定价值 | 任何 app 都能做。 | 需要银行语境或执行。 |
| 新加坡相关性 | 通用全球假设。 | 强本地语境。 |
| 人生结果影响 | 没有有意义结果。 | 保护或改善重大人生结果。 |
| 解释性 | 无法解释。 | 证据、不确定性和逻辑清楚。 |
| 用户控制 | 用户不能控制。 | 用户可编辑、批准、暂停或反转。 |
| 信任影响 | 可能降低信任。 | 通过透明增强信任。 |
| 执行价值 | 没有行动路径。 | 产生有用的已批准行动。 |
| 可扩展性 | 一次性 demo 逻辑。 | 可跨用户和目标复用。 |
| 技术可行性 | 现在不可行。 | 当前架构可构建。 |
| 监管可行性 | 可能被阻挡。 | 有清楚审查路径。 |
| 数据可用性 | 数据不可用。 | 数据可用或可安全 mock。 |
| 产品差异化 | 常见功能。 | 明显 FutureOS 优势。 |
| 长期有用性 | 只有 demo 价值。 | 多年有用。 |

## 阈值
- 通过：58 分或以上，且解释性、用户控制、信任、同意相关执行没有低于 3 分。
- 修订：42 到 57 分，或关键标准低于 3 分。
- 拒绝：低于 42 分，或违反产品宪法。
