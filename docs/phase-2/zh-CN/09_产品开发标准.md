# 产品开发标准

| 文档状态 | Phase 2 Bible draft |
| --- | --- |
| 版本 | 2.0.0-draft |
| 负责人 | FutureOS product, AI, design, banking, and engineering teams |
| 最后更新 | 2026-07-13 |

## 目的
定义创建 FutureOS 功能的完整运行流程。

## 范围
适用于从想法发现到实现、监控、复查和弃用。

## 产品信念
一个 FutureOS 功能不是因为看起来好就准备好了。它要解决真实用户问题、符合架构、尊重信任，并可审计。

## 必须出现的行为
- 每个功能必须有负责人、问题陈述、目标用户、预期结果、非目标、数据依赖、AI 角色、银行角色、风险级别、同意模型、成功指标、失败标准、回滚计划和文档更新。
- 每个功能必须经过产品、AI、用户、OCBC、新加坡、数据、同意、解释性、安全和衡量审查。

## 禁止出现的行为
- 不要只从 UI 灵感开始构建。
- 不要添加绕过三系统架构的功能。
- 不要发布没有解释的评分。
- 不要发布没有同意和回滚思考的行动流程。

## 设计影响
- 原型必须包含空状态、加载、成功、失败、恢复和偏好变化状态。
- 设计必须显示哪个系统拥有该功能。

## AI 影响
- AI 角色必须明确：解释、推荐、协商、批准后行动或 guardrail 运行。
- AI 输出必须结构化且可测试。

## 银行业务影响
- 必须记录 OCBC 相关性。
- 产品和服务执行必须可审计并有边界。

## 工程影响
- 实现前必须定义数据需求、schema、持久化、事件日志、权限和回滚。
- 文档更新是完成标准的一部分。

## 示例
- Guardian Memory 功能必须说明记忆事件 schema、同意边界、编辑删除规则和月度报告影响。
- Future Mirror 模拟必须说明输入字段、情景模型、分数解释和不确定性。

## 反模式
- Home 出现新卡片，但不连接 Life Graph、Future Mirror 或 Guardian。
- 功能使用 mock 数据却声称生产集成。
- CTA 视觉上能点但没有状态或历史。

## 决策检查清单
- [ ] 用户问题是否验证？
- [ ] 哪个系统拥有它？
- [ ] 需要什么数据？
- [ ] 需要什么同意？
- [ ] 如何衡量成功？
- [ ] 如何处理失败？

## Phase 2 影响
- Phase 2 添加更深 AI 或银行流程前必须使用此标准。

## 未来演进
- 该流程可演进成生产发现的轻量治理模型。

## 修订说明
- 文档是交付物，不是上线后清理任务。


## 开发阶段
| 阶段 | 审查问题 |
| --- |--- |
| 1. 问题发现 | 解决什么真实客户问题？ |
| 2. 证据审查 | 什么证据支持问题？ |
| 3. 产品适配评估 | 为什么属于 FutureOS？ |
| 4. 原则评估 | 哪些构建原则适用？ |
| 5. Life Graph 影响 | 客户理解发生什么变化？ |
| 6. Future Mirror 影响 | 模拟或取舍发生什么变化？ |
| 7. Guardian 影响 | 监控或行动发生什么变化？ |
| 8. OCBC 相关性 | 为什么需要银行？ |
| 9. 新加坡相关性 | 什么本地语境重要？ |
| 10. 数据需求 | 使用什么数据，为什么？ |
| 11. 同意需求 | 需要什么权限？ |
| 12. 解释性需求 | 如何解释输出？ |
| 13. 安全审查 | 可能出什么问题？ |
| 14. 原型 | 行为能否演示？ |
| 15. 用户测试 | 用户是否理解和信任？ |
| 16. 决策审查 | 记录什么决定？ |
| 17. 实现 | 是否在架构内构建？ |
| 18. 监控 | 有哪些日志和指标？ |
| 19. 上线后复查 | 是否改善结果？ |
| 20. 弃用或扩展 | 应继续、改变还是停止？ |
