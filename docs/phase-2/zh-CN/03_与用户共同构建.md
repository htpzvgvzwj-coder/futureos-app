# 与用户共同构建

| 文档状态 | Phase 2 Bible draft |
| --- | --- |
| 版本 | 2.0.0-draft |
| 负责人 | FutureOS product, AI, design, banking, and engineering teams |
| 最后更新 | 2026-07-13 |

## 目的
让 FutureOS 以用户问题为先，而不是以功能为先。

## 范围
适用于档案、目标、语言、可访问性、情景、恢复和用户测试。

## 产品信念
客户不是数据对象。FutureOS 必须通过尊重个人优先级、纠正、挫折和变化的人生阶段来建立关系。

## 必须出现的行为
- 没有验证过的用户问题就没有功能。
- 用户目标必须可编辑。
- 用户档案必须驱动输出。
- 支持自定义目标。
- 支持不同年龄、收入、关系、风险偏好和人生阶段。
- 尊重不同财务知识水平。
- 让每个分数可理解。
- 尽可能让每个行动可逆。
- 为不完整档案和冲突目标设计。

## 禁止出现的行为
- 假设单一人生旅程。
- 评判式语言。
- 强迫积极。
- 把忽略建议当成用户失败。
- 把输出锁定在固定人物。
- 隐藏建议变化原因。

## 设计影响
- 先用普通语言，再给技术细节。
- 显示档案和目标编辑控制。
- 为挫折后提供恢复路径。
- 让进度感觉像支持，而不是表演。

## AI 影响
- AI 只在必要时请求缺失语境。
- AI 应从用户纠正中学习。
- AI 不应对消费进行道德评判。

## 银行业务影响
- 银行支持应尊重客户声明的优先级。
- 产品匹配不能覆盖已声明目标。
- 客户经理升级应被呈现为支持，而不是销售压力。

## 工程影响
- 档案驱动输出必须用多个人物测试。
- 自定义目标需要灵活数据模型。
- 本地持久化要保存偏好，但不能假装是生产存储。

## 示例
- 如果 Karina 改变家庭规划优先级，Future Mirror 应更新情景影响。
- 如果用户跳过 guardrail，Guardian 可稍后提供更轻量恢复选项。

## 反模式
- 每个用户都收到同样的婚礼或购房模拟。
- 分数说纪律差却不解释计算。
- 被跳过的行动消失且没有恢复路径。

## 决策检查清单
- [ ] 用户问题是否验证？
- [ ] 用户能否编辑目标？
- [ ] 档案变化时输出是否变化？
- [ ] 语言是否尊重？
- [ ] 是否有恢复路径？

## Phase 2 影响
- Phase 2 应增加更丰富的自定义目标、用户纠正循环和关系记忆。
- 用户测试应覆盖不同人生阶段，而不只是 Karina。

## 未来演进
- FutureOS 应支持家庭或共享目标、暂时性挫折，以及多年变化的客户身份。

## 修订说明
- 关系通过行为体现：记住、解释、恢复和尊重同意。


## 功能发现模板
| 字段 | 必须回答 |
| --- |--- |
| 用户问题 | 这个功能解决什么真实客户困难？ |
| 用户分群 | 谁会在什么人生阶段遇到？ |
| 证据 | 什么观察、访谈、原型测试或银行模式支持？ |
| 当前替代方案 | 客户今天如何处理？ |
| 为什么当前方案失败 | 哪里混乱、手动、延迟或有风险？ |
| FutureOS 回应 | Life Graph、Future Mirror 或 Guardian 如何帮助？ |
| 风险 | 对信任、隐私或财务可能有什么问题？ |
| 成功指标 | 什么客户结果改善？ |
| 失败指标 | 什么信号说明功能伤害或困扰用户？ |
| 伦理考量 | 是否可能施压、羞辱或操控客户？ |
| OCBC 相关性 | 为什么银行特别适合帮助？ |
| 新加坡相关性 | 什么本地语境重要？ |
