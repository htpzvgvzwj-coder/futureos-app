# 术语表

| 文档状态 | Phase 2 Bible draft |
| --- | --- |
| 版本 | 2.0.0-draft |
| 负责人 | FutureOS product, AI, design, banking, and engineering teams |
| 最后更新 | 2026-07-13 |

## 目的
定义 FutureOS 术语，确保产品、AI、设计、银行业务和工程团队使用同一种语言。

## 范围
本术语表适用于所有 Phase 2 文档、原型流程、pitch 材料、产品决策记录和未来需求。

## 产品信念
一致语言保护架构。如果团队对同一责任使用不同词汇，功能会漂移并重复。

## 术语
| 术语 | 定义 |
| --- |--- |
| FutureOS | 嵌入 OCBC 风格移动银行体验里的 AI 原生人生管理层。 |
| AI 原生人生管理 | AI 持续理解档案、目标、风险、情景和已批准行动，以保护人生结果。 |
| 人生结果 | 有意义的客户结果，例如紧急资金韧性、家庭准备、购房准备、投资增长或退休安全。 |
| Life Graph | 理解客户人生阶段、目标、账户、产品、风险和财务行为的系统。 |
| Future Mirror | 在决策前模拟情景、取舍、未来后果和不确定性的系统。 |
| Future Self Guardian | 负责监控、建议、解释、准备行动，并只在同意边界内执行的 AI 代理。 |
| Goal Wallet | 用于保护已声明人生结果的目标化储蓄或分配容器。 |
| Goal readiness | 客户在当前条件下达成目标的准备程度。 |
| Future Score | 特定情景下长期财务健康的可解释指标。 |
| AI Confidence | 系统对分析的信心，必须和确定性分开。 |
| Product Fit Score | 产品类别支持已声明目标的适配指标。 |
| Guardian Memory | 帮助 Guardian 长期改善建议的决策和事件记忆。 |
| Guardian Reputation | 基于准确性、尊重同意、处理纠正、恢复成功和用户接受度的信任信号。 |
| Shared Goal Partnership | 用户拥有优先级和同意，Guardian 负责监控和解释的关系模型。 |
| Shared Responsibility | 有边界的运行模型，不是保证结果或替代判断。 |
| Autonomous Guardrails | 让 Guardian 在定义边界内行动的预批准限制。 |
| Autonomous Goal Lock | 计划中的 guardrail，用来保护目标不被冲突消费或产品决定削弱。 |
| Recovery Mode | 挫折后帮助恢复进度的 Guardian 状态。 |
| OCBC Product Matching | 以目标为先，把银行产品类别匹配到客户需求，并提供适配理由。 |
| Explainable AI | 显示证据、假设、不确定性和推荐逻辑的 AI 输出。 |
| Life event | 婚姻、家庭规划、购房、职业改变、创业或退休等重要变化。 |
| Financial DNA | 收入、消费、储蓄、风险偏好、目标和行为模式的原型概念。 |
| Decision memory | 决策、纠正、批准、拒绝和结果记录。 |
| Action Centre | 客户批准、修改、拒绝或复查 Guardian 行动的地方。 |
| Guardian Report | 展示进度、风险、行动、分数变化和建议的周期报告。 |
| Product orchestration | 围绕人生目标协调储蓄、支付、信贷、按揭、财富和保险。 |
| Live profile | 用于驱动输出的当前客户档案。 |
| Consent | 对数据使用、建议、行动准备或执行的明确许可。 |
| Reversibility | 在可能时撤销、暂停、修改或恢复行动的能力。 |

## 必须出现的行为
- 一致使用这些术语。
- 不要随意重命名顶层系统。
- 只有在能明确责任时才新增术语。

## 禁止出现的行为
- 不要把旧模块名作为竞争性顶层系统。
- 不要在没有同意语言时使用自主术语。
- 不要在没有解释时使用分数术语。

## 决策检查清单
- [ ] 术语是否已经存在？
- [ ] 术语是否澄清责任？
- [ ] 是否创造了重复模块？
- [ ] 是否需要产品决策记录？

## Phase 2 影响
- 新功能提案应引用术语表。
- 新核心术语应同时加入中英文文件。

## 未来演进
- 术语表应演进成设计、代码、分析和政策文档的受控词汇表。
