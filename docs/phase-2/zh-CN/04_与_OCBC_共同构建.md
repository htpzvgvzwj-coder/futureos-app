# 与 OCBC 共同构建

| 文档状态 | Phase 2 Bible draft |
| --- | --- |
| 版本 | 2.1.0-draft |
| 负责人 | FutureOS product, AI, design, banking, and engineering teams |
| 最后更新 | 2026-07-13 |

## 目的
定义 FutureOS 如何使用 OCBC 的银行优势，同时避免变成产品销售引擎。

## 范围
适用于产品匹配、账户语境、执行流程、客户经理升级、审计能力和跨产品协调。

## 产品信念
FutureOS 的价值来自把洞察连接到可信银行语境和已批准执行。没有 OCBC，它只是另一个外部 AI 顾问。

## 必须出现的行为
- 从客户结果开始，而不是从要销售的产品开始。
- 负责任地使用银行数据优势。
- 把推荐连接到适配理由。
- 围绕目标协调产品。
- 让产品冲突可见。
- 只有在客户理解理由后才准备申请。
- 审计产品推荐和执行步骤。

## 禁止出现的行为
- 没有适配理由就不推荐产品。
- 不能推荐会伤害受保护目标的产品。
- 生产前不能使用未验证产品声明。
- 不能隐藏产品优先级。
- 不能用恐惧销售保障或信贷。

## 设计影响
- 产品建议应作为目标里的支持路径出现，而不是广告。
- 显示产品类别为什么可能有帮助以及引入什么风险。
- OCBC 服务流程要冷静、专业。

## AI 影响
- AI 必须区分原型引用和已验证生产产品事实。
- AI 必须解释产品适配和产品冲突。
- AI 必须知道何时需要客户经理审查。

## 银行业务影响
- OCBC 提供账户语境、交易语境、执行优势、信任、监管、产品编排和客户经理升级。
- 类别包括存款、信用卡、支付、贷款与按揭、投资与财富、保险、客户经理支持。

## 工程影响
- 产品目录数据应和 AI prompt 分离。
- 推荐逻辑应支持政策检查和审计日志。
- 执行流程需要状态、同意、失败和回滚状态。

## 示例
- Goal Wallet 建议可以使用存款和自动转账支持已声明目标。
- 保险准备度可以建议 review，但不能声称某具体保单一定适合。

## 反模式
- FutureOS 从我们能卖哪个 OCBC 产品开始。
- 因客户储蓄高就推荐投资，却忽略紧急资金薄弱。
- 在解释负担能力前就推按揭。

## 决策检查清单
- [ ] 客户想达成什么结果？
- [ ] 为什么需要 OCBC？
- [ ] 相关产品类别是什么？
- [ ] 显示了什么适配理由？
- [ ] 可能伤害哪个受保护目标？

## Phase 2 影响
- 产品编排应放在 Guardian 和 Future Mirror 内部，而不是新顶层系统。
- 原型产品引用在验证前必须标记为原型。

## 未来演进
- 生产版本需要产品、法律、风险、合规和数据审查。

## 修订说明
- 银行优势必须服务客户目标，而不是替代客户目标。


## 核心规则
FutureOS 永远不从“我们能卖哪个产品”开始。

它从“客户想达成什么结果”开始。


## 产品冲突规则
FutureOS 必须知道什么时候不能推荐某个产品。银行级产品引擎的价值，既来自匹配，也来自克制。

| 冲突条件 | 产品规则 | 必须解释 |
| --- | --- | --- |
| 紧急资金低于目标 | 不要优先推荐高风险投资加码。 | 解释流动性先保护近期韧性，再谈增长。 |
| 目标期限少于 12 个月 | 不要把高波动资产作为主要资金来源。 | 解释短期限和本金保护需求。 |
| 信用卡或无担保债务较高 | 不要把新增消费信贷作为第一方案。 | 解释债务压力和还款优先级。 |
| 存在保险缺口 | 不要自动推荐最高保费产品。 | 解释保障需求、可负担性和复查要求。 |
| 按揭准备度不足 | 不要在改善负担能力前推动贷款申请。 | 解释偿债能力、首付和稳定性差距。 |
| 产品利益和受保护目标冲突 | 不要隐藏冲突。 | 显示支持目标、受损目标和替代选项。 |
| 客户反复拒绝类似建议 | 不要不加修改地重复推荐。 | 解释什么变化了，或改为轻量干预。 |
| 适配性数据不足 | 不要把产品说成适合。 | 请求缺失信息或升级人工审查。 |

## 产品适配证据
每个产品推荐都必须显示证据栈。只有系统能回答以下字段，产品匹配才算完成。

| 证据字段 | 必须回答 |
| --- | --- |
| 支持的目标 | 这个产品类别支持哪个客户目标？ |
| 使用的客户数据 | 哪些档案、账户、交易、目标或偏好影响了推荐？ |
| 适配理由 | 为什么这个产品类别现在相关？ |
| 产品风险 | 可能有什么风险、成本、锁定、波动、债务或流动性问题？ |
| 考虑过的替代方案 | 是否考虑过低风险、低成本或非产品替代方案？ |
| 产品冲突检查 | 检查了哪些受保护目标？ |
| 预期影响 | 如果客户接受，目标会发生什么变化？ |
| 限制 | 什么未知或不能保证？ |
| 人工审查要求 | 是否需要 Relationship Manager、持牌顾问或政策审查？ |

## OCBC 产品编排流程
产品编排必须发生在三系统架构内部。

| 步骤 | 所属系统 | 输出 |
| --- | --- | --- |
| 检测需求 | Life Graph | 检测需求、证据、紧急程度、受影响目标。 |
| 模拟影响 | Future Mirror | 产品支持前后的情景影响。 |
| 检查冲突 | Future Self Guardian | 受保护目标冲突检查和流动性检查。 |
| 匹配类别 | Future Self Guardian | 产品类别，而不是未验证产品声明。 |
| 解释适配性 | Future Self Guardian | 证据栈、限制、替代方案。 |
| 请求同意 | Future Self Guardian | 批准准备、申请、预约或执行。 |
| 必要时升级 | Relationship Manager 或持牌审查 | 带理由和摘要的人工交接。 |
| 监控结果 | Guardian Memory 和月度报告 | 产品路径是否帮助目标。 |

## 产品推荐状态
| 状态 | 含义 | 允许行动 |
| --- | --- | --- |
| Not applicable | 产品类别不支持目标。 | 不显示为推荐。 |
| Watch | 产品未来可能相关。 | 只显示为未来准备提醒。 |
| Explore | 产品类别可能有帮助，但数据不完整。 | 请求更多语境或显示教育内容。 |
| Prepare | 客户看起来可以进入复查。 | 准备 checklist 或预约。 |
| Recommend review | 产品类别相关，但需要人工或政策审查。 | 提供 Relationship Manager 或持牌审查。 |
| Ready for consent | 行动清晰、低风险且符合规则。 | 请求批准。 |
| Blocked | 冲突、缺少同意或风险阻止推荐。 | 解释原因并显示替代方案。 |

## 客户关系维护
OCBC 应使用 FutureOS 维护信任，而不只是增加产品持有。

- 如果 FutureOS 因为会伤害受保护目标而不推荐产品，这种克制应该可见。
- 如果建议需要人工审查，交接应保留语境，让客户不用重复完整故事。
- 如果客户拒绝产品路径，FutureOS 应记住原因，并在条件改变前避免重复推动。
- 如果客户采用产品，FutureOS 必须监控它是否继续支持原始目标。
- 如果人生变化导致产品不再适合，FutureOS 必须提示复查，而不是静默保留在计划里。

## 产品匹配工程合同
- 产品目录、资格、风险标签和适配规则必须数据驱动并版本化。
- 产品匹配逻辑不能只嵌在 prompt 文本里。
- 每个产品推荐事件必须包含适配证据、冲突结果和同意状态。
- 原型产品引用必须带有 `prototype_reference` 标记。
- 生产产品推荐需要已验证产品数据、政策审查和审计日志。
- 产品推荐 UI 必须支持 `not recommended` 和 `blocked` 状态，而不只是正向匹配。
