# 产品决策记录

| 文档状态 | Phase 2 Bible draft |
| --- | --- |
| 版本 | 2.0.0-draft |
| 负责人 | FutureOS product, AI, design, banking, and engineering teams |
| 最后更新 | 2026-07-13 |

## 目的
为 FutureOS 创建可复用的决策记忆系统。

## 范围
适用于架构变化、功能接受、自主性、评分、数据使用、产品逻辑和顶层系统边界。

## 产品信念
严肃的 AI 银行产品需要记住为什么做出决定，而不只是记住发布了什么。

## 必须出现的行为
- 架构决定、新顶层系统、自主性变化、产品推荐逻辑、分数计算、新数据类别和执行流程都要记录。
- 把决策记录连接到受影响原则和复查日期。

## 禁止出现的行为
- 不要静默改变核心架构。
- 没有决策记录批准不要增加第四个顶层系统。
- 不要把决策理由只留在聊天或 presentation 里。

## 设计影响
- 决策结果应影响 UI 语言和层级。
- 被拒绝决定也要记住，避免重复。

## AI 影响
- AI 行为变化必须包含风险、可逆性、信心和日志决定。

## 银行业务影响
- 产品、执行和升级决定必须包含银行影响和审计需求。

## 工程影响
- 决策记录应存放在 Git 中并被功能文档引用。
- 实现可在相关位置链接决策 ID。

## 示例
- PDR-001 保护三系统架构。
- PDR-008 要求所有分数解释计算和含义。

## 反模式
- 团队因忘记旧理由而重新把 Goal Negotiator 做成新 tab。
- 分数公式变化但没有日志或复查。

## 决策检查清单
- [ ] 是否改变架构？
- [ ] 是否影响同意？
- [ ] 是否影响 AI 自主性？
- [ ] 是否影响产品推荐？
- [ ] 是否设置复查日期？

## Phase 2 影响
- Phase 2 必须用决策记录防止功能膨胀。

## 未来演进
- 如果 FutureOS 走向生产，决策记录可演进为正式治理记录。

## 修订说明
- 初始记录集捕捉 Phase 1 优化期间已经形成的决定。


## 模板
| 字段 | 说明 |
| --- |--- |
| PDR ID | 唯一编号。 |
| 标题 | 决策标题。 |
| 日期 | 决策日期。 |
| 状态 | 草案、接受、取代、拒绝。 |
| 负责人 | 决策负责人。 |
| 背景 | 背景信息。 |
| 问题 | 正在解决的问题。 |
| 考虑选项 | 审查过的替代方案。 |
| 决定 | 选择路径。 |
| 理由 | 为什么选择此路径。 |
| 受影响原则 | 构建原则和宪法规则。 |
| 用户影响 | 对客户的影响。 |
| 银行影响 | 对 OCBC 流程的影响。 |
| AI 影响 | 对模型或代理行为的影响。 |
| 数据影响 | 需要或改变的数据。 |
| 安全影响 | 安全或隐私影响。 |
| 风险 | 已知风险。 |
| 可逆性 | 回滚或迁移路径。 |
| 成功标准 | 如何验证决定。 |
| 复查日期 | 何时重新审查。 |
