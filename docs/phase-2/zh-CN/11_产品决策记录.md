# 产品决策记录

| 文档状态 | Phase 2 Bible draft |
| --- | --- |
| 版本 | 2.0.0-draft |
| 负责人 | FutureOS product, AI, design, banking, and engineering teams |
| 最后更新 | 2026-07-21 |

## 目的
为 FutureOS 创建可复用的决策记忆系统。

## 范围
适用于架构变化、功能接受、自主性、评分、数据使用、产品逻辑和顶层系统边界。

## 产品信念
严肃的 AI 银行产品需要记住为什么做出决定，而不只是记住发布了什么。

## 必须出现的行为
- 架构决定、新顶层系统、自主性变化、产品推荐逻辑、分数计算、新数据类别和执行流程都要记录。
- 把决策记录连接到受影响原则和复查日期。

## 禁止出现的行为
- 不要静默改变核心架构。
- 没有决策记录批准不要增加第四个顶层系统。
- 不要把决策理由只留在聊天或 presentation 里。

## 设计影响
- 决策结果应影响 UI 语言和层级。
- 被拒绝决定也要记住，避免重复。

## AI 影响
- AI 行为变化必须包含风险、可逆性、信心和日志决定。

## 银行业务影响
- 产品、执行和升级决定必须包含银行影响和审计需求。

## 工程影响
- 决策记录应存放在 Git 中并被功能文档引用。
- 实现可在相关位置链接决策 ID。

## 示例
- PDR-001 保护三系统架构。
- PDR-008 要求所有分数解释计算和含义。

## 反模式
- 团队因忘记旧理由而重新把 Goal Negotiator 做成新 tab。
- 分数公式变化但没有日志或复查。

## 决策检查清单
- [ ] 是否改变架构？
- [ ] 是否影响同意？
- [ ] 是否影响 AI 自主性？
- [ ] 是否影响产品推荐？
- [ ] 是否设置复查日期？

## Phase 2 影响
- Phase 2 必须用决策记录防止功能膨胀。

## 未来演进
- 如果 FutureOS 走向生产，决策记录可演进为正式治理记录。

## 修订说明
- 初始记录集捕捉 Phase 1 优化期间已经形成的决定。

## PDR-013 详情:Life Graph 数据源 Provider 抽象层

| 字段 | 内容 |
| --- | --- |
| PDR 编号 | PDR-013 |
| 标题 | Life Graph 通过可插拔的 provider 注册表读取客户数据,而不是写死单一数据源。 |
| 日期 | 2026-07-21 |
| 状态 | 草案 —— 尚未经产品/合规负责人评审,不能标记为"接受"。此记录由 AI 助手根据现有代码库和"Build For The Future"原则起草,尚未经过法务或银行治理审查。 |
| 负责人 | 未指定 —— 需要指定一位 FutureOS 产品负责人。 |
| 背景 | Life Graph 现在真正的数据源只有一个:`preferences.profile`,客户自己填写、只存在浏览器 localStorage 里的普通对象。尽管产品叙事和 `CrossBankDataScreen` 里用了"SGFinDex"式跨行数据的说法,这个原型从来没有真正接入过任何外部银行或聚合平台——那些屏幕明确标注为使用模拟数据的概念预览。`app/page.jsx` 里已经建了一层 provider 抽象(`LIFE_GRAPH_PROVIDERS` 注册表,目前只有一个条目 `manualEntryProvider`,接口形状为 `{ id, labelKey, getProfile(preferences), getCustomGoals(preferences) }`),专门是为了让第二个真实数据源能够接入,而不需要重写现在约 20 处通过 `getUserProfile()`/`getCustomGoals()` 读取数据的调用点。 |
| 问题 | 要真正注册一个第二个真实 provider(比如真正的 SGFinDex 式聚合,或某家银行的开放银行 API),在技术、法律、架构上到底需要什么,同时不能违反现有的宪法原则(数据责任先于个性化、执行前需同意、可审计性、"AI 不能编造产品事实")? |
| 考虑过的选项 | (1) 直接接入 SGFinDex —— 新加坡真正的金融数据交换平台;需要 MAS 认可的数据共享渠道和 OCBC 一方的真实协议,这不是一个原型项目能拿到的。(2) 商用开放银行聚合平台(如果新加坡有类似 Plaid 的服务)——比逐家银行对接门槛更低,但仍需要真实商业协议和 API key。(3) 用某个真实聚合平台公开的沙盒/演示环境做第二个 provider——数据是真正外部的,但明确标注为非生产环境,不需要真实银行生产凭证就能把整条架构跑通验证。(4) 在真正的合作关系出现之前,继续只用手动录入,把这个 provider 注册表当作纯架构上的准备(截至本 PDR 起草时的现状)。 |
| 决策 | 尚未决定 —— 本 PDR 的作用是把需求和现有缺口记录清楚,方便未来有指定负责人评审后,把选项(3)(沙盒模式的第二个 provider)当作风险最低的下一步来评估。起草这份 PDR 本身不代表任何代码改动。 |
| 理由 | 一个黑客松/原型项目拿不到真正的生产级银行数据共享协议,所以任何没有选项 (1) 或 (2) 支撑的"已接入第二个真实数据源"的说法,本身就是编造的产品事实——这正是宪法明确禁止的。沙盒模式接入是唯一既真正外部、又在没有真实合作关系时依然诚实可得的选项。 |
| 涉及的原则 | Build With OCBC(有意义的功能必须连接真实银行数据,而不是没有银行优势的外部顾问式行为);数据责任先于个性化;AI 不能编造产品事实;可审计性是客户体验的一部分。 |
| 对用户的影响 | 目前没有影响——在真正接入某个 provider 之前不会有任何行为变化。如果真的接入,客户需要一个明确的、可撤销的同意流程才能关联外部账户——这个流程现在整个 App 里都不存在(目前完全没有 auth/同意授权体系,跟 [[futureos_differentiation_strategy]] 里提到的多用户缺口是同一个问题)。 |
| 对银行的影响 | 真正接入之后,OCBC 会在 Guardian 的推荐逻辑里展示并基于另一家机构的数据采取行动——通过这种方式获得的每一个数字都必须明确标注来源(见"对数据的影响"),这样银行经理在复核升级案例时,能一眼分辨哪些是 OCBC 自己确认的数据、哪些是外部机构报告的数据。 |
| 对 AI 的影响 | Guardian 现在"AI 不碰数字"的纪律,默认所有输入要么是客户自己声明的资料,要么是 OCBC 自己确认的储蓄计划记录,完全没有"数据来源"或"数据新鲜度"的概念。真正接入第二个 provider 需要扩展这条纪律:每一个计算出来的数字都要带上 `source` 标签(来自哪个 provider)和 `asOf`/`lastSyncedAt` 时间戳,这样 Guardian 就不会把一个三天前从外部拉取的余额,当成客户刚刚确认的实时数字来对待。 |
| 对数据的影响 | 新增的数据类别:其他机构的账户余额/交易记录、只读的 OAuth 式访问令牌、同意范围和有效期、同步时间戳。这些在现有 schema(`scripts/migrate.sql`)里完全没有建模——真正的 provider 需要自己的令牌存储表,要加密存储,并且要有明确的撤销路径,而不是简单扩展 `preferences.profile`。 |
| 对安全的影响 | 任何外部获取的个人财务数据都要符合新加坡 PDPA(个人数据保护法);令牌要加密存储;不能持久化第三方原始凭证;要有真正可用的撤销授权流程(现在的应用数据模型完全没有"断开某个数据源"这个概念——`deleteLocalData()` 只会清空这台浏览器的本地状态)。 |
| 风险 | 监管风险(MAS/PDPA);如果第三方数据陈旧或错误、Guardian 却据此行动,会有 OCBC 品牌/责任风险;Guardian 计算链路里加入实时网络调用会增加延迟;如果用商用聚合平台会有持续成本。 |
| 可回退性 | 在架构层面很高——一个 provider 只是 `LIFE_GRAPH_PROVIDERS` 里的一条记录;移除它不需要改动下游约 20 个调用点。但在数据层面,一旦真正存过外部令牌/数据,可回退性就低得多——真正的撤销并删除路径必须在第二个 provider 上线之前就存在,不能等以后再补。 |
| 成功标准 | 第二个 provider 只需要实现现有的 `{ getProfile, getCustomGoals }` 接口(按上面"对 AI 的影响"扩展成异步 + 带来源/新鲜度标记),不需要改动任何现有调用点——一旦真的有这样一个 provider,这一点是可以机械验证的。 |
| 复审时间 | 不按日期定——等真正开始讨论数据共享合作(SGFinDex、某个聚合平台,或某家具体的合作银行)时再复审,而不是按日历。 |
