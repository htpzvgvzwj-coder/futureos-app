# 关系与共同责任

| 文档状态 | Phase 2 Bible draft |
| --- | --- |
| 版本 | 2.1.0-draft |
| 负责人 | FutureOS product, AI, design, banking, and engineering teams |
| 最后更新 | 2026-07-13 |

## 目的
定义 FutureOS 随时间与用户建立信任的关系模型。

## 范围
适用于 Guardian 行为、用户权利、同意、记忆、恢复、分歧、表现报告和升级。

## 产品信念
关系不是靠情绪化文案创造的。关系来自系统记住、解释、保护、承认不确定性并尊重客户控制。

## 必须出现的行为
- 使用 Shared Goal Partnership 模型。
- 让用户权利可见。
- 让 Guardian 义务可见。
- 通过行为追踪 Guardian 声誉。
- 支持权限渐进。
- 支持承认错误和修订策略。
- 支持冲突、恢复模式、目标放弃、关系重置、撤回同意和人工升级。

## 禁止出现的行为
- 共同责任不等于 AI 法律责任、保证结果、强制储蓄、情感依赖、操控或 AI 替代客户判断。
- 不要用情感语言隐藏不确定性。
- 不要因为用户不同意而惩罚用户。

## 设计影响
- 通过合同、记忆、报告和行动历史展示关系。
- 使用冷静状态：观察中、监控中、有风险、恢复中、等待批准。
- 在设置中显示重置和撤回。

## AI 影响
- Guardian 必须记住决策，而不只是聊天。
- Guardian 必须在用户优先级变化时修订策略。
- Guardian 必须承认先前信心估计过高的情况。

## 银行业务影响
- OCBC 通过让 Guardian 可问责、可审计并受银行规则约束来维护信任。
- 客户经理升级应被呈现为关怀连续性。

## 工程影响
- 分别存储同意、纠正、行动和记忆事件。
- 构建支持月度报告和关系声誉的事件日志。
- 支持暂停和撤回状态。

## 示例
- 如果 Karina 拒绝消费 guardrail，Guardian 记录原因并稍后提出更轻量提醒。
- 如果客户撤回同意，自主行动停止；若允许，只读规划可以继续。

## 反模式
- Guardian 说我知道什么最好然后执行。
- 应用隐藏被拒绝建议。
- AI 表现失望或制造内疚。

## 决策检查清单
- [ ] 用户是否拥有优先级？
- [ ] Guardian 是否负责监控和解释？
- [ ] 同意能否撤回？
- [ ] 分歧能否记录？
- [ ] 是否支持恢复？

## Phase 2 影响
- Phase 2 应增加共享目标合同、Guardian 声誉指标和关系重置流程。

## 未来演进
- 长期关系可以支持家庭或共享目标，但个人同意和边界仍必须存在。

## 修订说明
- 这是主要设计创新层：FutureOS 不只是聪明，而是关系上可问责。


## 共享目标伙伴关系
| 用户负责 | Guardian 负责 |
| --- |--- |
| 优先级 | 持续规划 |
| 最终决定 | 监控 |
| 同意 | 风险检测 |
| 个人价值观 | 恢复规划 |
| 目标变化 | 解释 |
| 权限边界 | 已批准执行 |
| 暂停或撤回权利 | 决策记忆和报告 |

## 关系维护机制
- 共享目标合同：每个重大目标都有用户与 Guardian 的可见协议。
- 信任渐进：Guardian 从解释开始，通过准确和尊重的行为赢得权限。
- Guardian 声誉：由被接受建议、被纠正假设、避免风险、恢复成功和尊重同意衡量。
- 月度关系报告：显示 Guardian 做了什么、什么变化、哪里判断错误、需要复查什么。
- 恢复模式：在挫折后帮助用户，而不是放弃目标。
- 关系重置：客户可以重启偏好、权限或目标，而不是删除整个账户故事。


## Guardian Responsibility Ledger
共同责任必须可检查。每个受保护目标都应该拥有 Guardian Responsibility Ledger。

| Ledger 字段 | 含义 |
| --- | --- |
| Goal Mission | 用客户语言写出的被保护人生结果。 |
| Guardian Obligations | Guardian 必须监控、解释、复查和准备什么。 |
| Customer Obligations | 客户同意负责决定、更新、批准或披露什么。 |
| Authorised Actions | Guardian 在当前权限级别下可准备或执行的行动。 |
| Review Frequency | Guardian 必须多常复查目标。 |
| Risk Thresholds | 触发提醒、恢复或升级的条件。 |
| Recovery Trigger | 什么算偏移、错过目标、超支、进度停滞或人生语境变化。 |
| Performance Target | Guardian 被期望改善或保护什么。 |
| Current Status | Observing、on track、at risk、recovery、paused、completed 或 abandoned。 |
| Decision History | 批准、拒绝、纠正、计划变化、升级和结果。 |

## Shared Goal Contract
Shared Goal Contract 不是法律合同，而是让关系显性化的产品机制。

| 合同部分 | 必须内容 |
| --- | --- |
| Goal statement | 客户想要什么，以及为什么重要。 |
| Priority level | Critical、important、flexible、experimental 或 paused。 |
| Protected boundaries | 没有明确复查时不应被牺牲的边界。 |
| Guardian role | Guardian 会监控和建议什么。 |
| Customer role | 客户必须批准、更新或决定什么。 |
| Autonomy level | 该目标当前 AI 独立级别。 |
| Review rhythm | 每周、每月、每季度或事件触发。 |
| Escalation path | 何时需要 Relationship Manager 或人工审查。 |
| Exit condition | 什么会完成、暂停、重置或放弃目标。 |

## Shared Goal Contract 状态
| 状态 | 含义 | Guardian 义务 |
| --- | --- | --- |
| Draft | 目标存在，但数据或同意不足。 | 只请求必要缺失语境。 |
| Proposed | Guardian 已建议计划。 | 解释证据、假设和取舍。 |
| Active | 客户接受计划。 | 监控、报告和检测偏移。 |
| At Risk | 阈值被越过。 | 提醒、解释影响并生成恢复选项。 |
| Recovery | 客户批准或请求修订计划。 | 追踪恢复进度并调整策略。 |
| Paused | 客户暂停目标或撤回权限。 | 停止行动并保留历史。 |
| Completed | 目标结果达成。 | 生成完成复查和下一步选项。 |
| Abandoned | 客户主动停止追求目标。 | 不评判地记录原因并保护剩余目标。 |

## Trust Event Model
Guardian 的信任必须因为行为变化，而不是因为文案变化。

| 事件 | 信任影响 | 必须回应 |
| --- | --- | --- |
| 建议被接受且结果改善 | 提升该目标类别信任。 | 记录成功并解释什么有效。 |
| 建议被接受但结果中性 | 不自动提升信任。 | 继续监控并解释不确定性。 |
| 建议被拒且有原因 | 不惩罚客户，Guardian 学习。 | 保存原因并修订未来策略。 |
| Guardian 提前发现风险 | 如果提醒有用且及时，提高信任。 | 记录提前量和结果。 |
| Guardian 错过重大风险 | 降低信任并进入复查。 | 承认遗漏、解释原因、提出预防。 |
| Guardian 推荐不适合或冲突产品 | 明显降低信任。 | 移除推荐、升级审查、修订规则。 |
| Guardian 尊重同意边界 | 维持或提升信任。 | 记录边界并显示客户控制。 |
| Guardian 违反同意边界 | 严重信任失败。 | 停止自主性、通知客户、升级审查。 |

## 权限渐进
Guardian 不应该默认获得 Level 5 自主权。权限必须被赢得。

| 权限阶段 | 解锁要求 | 允许行为 |
| --- | --- | --- |
| Explain only | 客户启用分析。 | 解释证据和可能结果。 |
| Recommend | 档案和目标足够完整。 | 推荐选项但不执行。 |
| Negotiate | 客户至少有两个竞争性活跃目标。 | 显示取舍和平衡计划。 |
| Act with approval | 客户接受过之前建议，或明确授权准备行动。 | 准备行动，只在批准后执行。 |
| Autonomous guardrails | 表现良好、明确限制、无同意违规、数据充足且客户批准。 | 只在限制内执行预批准行动。 |

更高自主性的资格需要：观察到良好表现、用户批准、没有同意违规、足够财务数据，以及清楚的可逆性或安全限制。

## 用户拒绝建议时
重复拒绝不是不听话，而是产品数据。

- 第一次拒绝：询问可选原因，并提供更轻量替代方案。
- 第二次类似拒绝：降低推荐频率，并检查目标优先级是否变化。
- 第三次类似拒绝：把策略移入复查，并询问目标、预算、语气或自主级别是否需要改变。
- 负面体验后的拒绝：进入信任修复模式，并解释 Guardian 会改变什么。
- 价值观不匹配导致拒绝：更新用户偏好，不把决定说成财务上不理性，除非风险严重。

## 承认错误与信任修复
Guardian 错误时必须有恢复协议。

1. 用普通语言说清错误。
2. 解释哪个证据或假设导致错误。
3. 说明影响了什么。
4. 显示是否采取过行动。
5. 在可能时停止或反转行动。
6. 必要时降低受影响目标的自主性。
7. 修订记忆、规则或阈值。
8. 高影响案例提供人工审查。
9. 把事件记录在 Guardian History 和月度报告。

## 责任边界
共同责任不模糊责任。

| 领域 | 客户责任 | Guardian 责任 | OCBC 责任 |
| --- | --- | --- | --- |
| 优先级 | 决定什么最重要。 | 记住并尊重优先级顺序。 | 提供安全的优先级管理系统。 |
| 同意 | 批准、拒绝、暂停或撤回。 | 永不超过当前权限。 | 执行同意边界和审计日志。 |
| 数据准确性 | 纠正已知档案错误。 | 标记缺失或不一致数据。 | 保护数据并说明限制。 |
| 建议 | 决定是否接受。 | 解释证据、替代方案和风险。 | 确保推荐框架负责任。 |
| 执行 | 批准重大行动。 | 只在批准限制内执行。 | 提供安全可审计执行通道。 |
| 恢复 | 重新确认目标或限制。 | 生成恢复选项。 | 支持升级和服务连续性。 |
