# Phase 2 路线图

| 文档状态 | Phase 2 Bible draft |
| --- | --- |
| 版本 | 2.0.0-draft |
| 负责人 | FutureOS product, AI, design, banking, and engineering teams |
| 最后更新 | 2026-07-13 |

## 目的
把 Phase 2 组织成深化 FutureOS 的工作流，而不是复制模块。

## 范围
适用于计划中的产品、AI、设计、银行、本地化、同意和衡量工作。

## 产品信念
Phase 2 应深化核心系统和关系模型，而不是追逐无关功能。

## 必须出现的行为
- 清楚区分已存在、原型、计划中、需要研究和未来概念。
- 每个工作流定义目标、当前状态、目标状态、依赖、交付物、风险、成功指标和退出标准。

## 禁止出现的行为
- 不要把计划中功能说成已完成。
- 不要创建新顶层系统。
- 不要隐藏研究或合规依赖。

## 设计影响
- 路线图工作要映射到可见用户流程。
- 关系和问责要被设计成功能，而不是文字。

## AI 影响
- 代理行为应从解释推进到受治理的自主。
- 目标依赖模拟应替代静态输出。

## 银行业务影响
- OCBC 产品编排必须保持目标优先和适配意识。
- 执行流程在生产声明前需要审查。

## 工程影响
- 路线图需要档案、目标、情景、行动、同意、记忆和报告 schema。
- 衡量和日志是工作流，不是事后补充。

## 示例
- Guardian Memory 根据当前构建状态标记为计划中或原型，而不是生产记忆。
- Autonomous Goal Lock 是计划中，需要同意审查。

## 反模式
- 只列屏幕没有依赖的路线图。
- 把 localStorage 说成生产数据架构。
- 验证前加入产品名称。

## 决策检查清单
- [ ] 状态标签是否准确？
- [ ] 所属系统是否清楚？
- [ ] 风险是否命名？
- [ ] 退出标准是否可衡量？
- [ ] 依赖是否可见？

## Phase 2 影响
- 本文件就是 Phase 2 执行地图。

## 未来演进
- 之后的路线图可把工作流转成带 PDR 引用的发布计划。

## 修订说明
- 状态标签故意保持保守。


## 状态标签
- 已存在：当前原型可见。
- 原型：以 mock 或本地行为存在，不是生产集成。
- 计划中：Phase 2 计划构建。
- 需要研究：需要发现、政策、数据或技术验证。
- 未来概念：Phase 2 之后的有用方向。
