# 为未来而构建

| 文档状态 | Phase 2 Bible draft |
| --- | --- |
| 版本 | 2.0.0-draft |
| 负责人 | FutureOS product, AI, design, banking, and engineering teams |
| 最后更新 | 2026-07-13 |

## 目的
确保 FutureOS 在黑客松之后仍然有用，并能跨人生阶段、产品、模型和法规演进。

## 范围
适用于数据架构、模型替换、产品演进、法规、渠道、共享目标、审计历史、迁移和可逆性。

## 产品信念
FutureOS 是关系型产品。它必须超越单一 demo、单一人物、单一 LLM provider 和单一情景模板。

## 必须出现的行为
- 支持长期目标。
- 支持人生阶段变化。
- 支持产品演进。
- 支持模型替换。
- 支持可扩展数据架构。
- 支持新法规。
- 支持人机协作。
- 支持区域扩展。
- 支持家庭和共享目标。
- 支持审计历史。
- 支持迁移和可逆。

## 禁止出现的行为
- 不要把架构绑定单一人物。
- 不要假设单一 LLM provider。
- 不要把固定产品名写进业务逻辑。
- 不要把固定情景模板当成产品真相。
- 目标变化时不要删除旧决策历史。

## 设计影响
- 设计长期连续性，而不是一次性屏幕。
- 显示历史、进度、恢复和复查时刻。
- 让未来变化感觉像关系演进，而不是重置。

## AI 影响
- AI 记忆必须保存持久决策和纠正。
- 模型输出需要足够可复现以便审计。
- AI 应把持久产品规则和可替换模型推理分离。

## 银行业务影响
- 产品会改变，客户结果会持续。
- FutureOS 应把产品演进连接到目标连续性。
- 合规变化必须被吸收，而不破坏客户信任。

## 工程影响
- 目标、情景、行动、权限、产品和记忆需要稳定抽象。
- 避免用字符串做业务逻辑。
- 提供数据迁移和弃用路径。

## 示例
- 客户停止购房规划时，不应丢失紧急资金或退休历史。
- 投资产品名变化时，目标策略仍应可理解。

## 反模式
- 所有情景逻辑都只和婚礼有关。
- Guardian Memory 只存成 UI 文字。
- 模型升级后建议变化却没有解释。

## 决策检查清单
- [ ] 能否适应新人物？
- [ ] 模型能否替换？
- [ ] 产品名能否变化？
- [ ] 用户能否反转或迁移？
- [ ] 历史是否可审计？

## Phase 2 影响
- Phase 2 即使仍使用 mock 或本地数据，也应引入持久的目标、行动、记忆和同意 schema。

## 未来演进
- 未来时间层应支持即时、近期、中期和长期决策。

## 修订说明
- 面向未来的架构本身就是信任功能。


## 时间跨度
| 跨度 | 范围 | FutureOS 责任 |
| --- |--- |--- |
| 即时 | 0-12 个月 | 消费、紧急资金、行动提醒和短期目标纪律。 |
| 近期 | 1-3 年 | 购房准备、家庭规划、债务控制和投资习惯。 |
| 中期 | 3-10 年 | 教育、职业转换、创业、按揭策略和保障规划。 |
| 长期 | 10-30 年 | 退休安全、财富增长、传承规划和重大人生阶段转换。 |
